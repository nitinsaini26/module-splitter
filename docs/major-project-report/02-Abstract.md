# Abstract

Large code files are hard to understand and harder to maintain. A single file may contain a React component, a few custom hooks, several utility functions, and TypeScript types. Even if the code works, this structure creates long functions, complex imports, and fragile test setups. The project in this repository, ASTra v3 Module Splitter, targets that everyday problem in a practical way. It is a VS Code extension that analyzes a TypeScript or JavaScript module and produces a clean split plan. The plan shows which regions should be moved into new files, which imports are required, and how the updated source should look after extraction.

ASTra v3 follows an eight stage pipeline. It starts with AST parsing using the TypeScript Compiler API, which gives a reliable view of the code structure. Each top level declaration is converted into a region with metadata such as name, kind, line range, and symbol usage. A dependency graph is then built between regions, and classic graph algorithms are applied to find cycles and order extraction. The tool computes a set of metrics per region and per file, including cyclomatic complexity, cognitive complexity, Halstead volume and effort, maintainability index, and a custom testability score. It also runs more than twenty code smell rules for React and TypeScript and can extend those rules to Vue, Angular, and Svelte through framework plugins.

The decision to extract a region is made by an Extraction Oracle. Instead of a single hard rule, the Oracle combines multiple signals, such as size pressure, complexity, coupling, smell severity, and kind affinity. A calibrated threshold based on Halstead effort adjusts the extraction aggressiveness for each file. This helps avoid over splitting trivial files and under splitting complex files. When extraction is approved, the tool resolves all required imports, generates file content, produces a barrel index, and optionally prepares test stubs. Finally, it builds a linkage map so the user can see how the new files will depend on each other.

ASTra is designed to be safe for daily use. The apply step uses a single WorkspaceEdit so the entire refactor can be undone with one undo action. It provides a webview with eight tabs to review regions, metrics, smells, candidates, and generated files. Inline diagnostics show code smells directly in the editor, and a status bar grade gives a quick health indicator. Incremental caching avoids re analyzing unchanged regions, which makes repeated runs faster.

This report documents the project in simple language. It explains the motivation, the tools and technologies used, the algorithms that power the pipeline, and the overall methodology. It also provides observations and discusses results in terms of code quality, usability, and limitations. The report concludes with future work based on the current roadmap and accuracy improvements that can further strengthen the system.
